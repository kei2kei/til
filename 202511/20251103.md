## 勉強になったこと

- Reactベーシック
  - ReactのUIの考え方
    - それぞれのパーツ（ナビゲーション、ヘッダー）などをブロックと考える（コンポーネント）
    - 必要なところだけに更新をかけにいく
- コンポーネントとは？
    - UIを返す関数
    - しかし、HTMLと区別するために大文字から始め、使う時はHTML同様に<>で括る必要がある
- SSR/CSR
    - それぞれに長所があるSSRはコードを送る量も減るので効率がいいがインタラクティブなUIの実現ができない
    - インテラクティぶな反応を得るべき箇所を限定的にコンポーネント化し最小にすることでどちらの恩恵も得ることが可能
- propsについて
    - コンポーネントで持っているstateは同じコンポーネントを複数使っていてもそれぞれ独立している
    - [もし、そのコンポーネント間で値の更新などを共有したい場合は一つ上の階層でpropsを定義し、渡さなければならない](https://react.dev/learn#sharing-data-between-components)

- アプリ作成時のポイント
    - まずはデザインからコンポーネントになりそうな部分を推測してみる
        - 概ねデザイナーがページのサンプルを持っているので、そこからコンポーネントになりそうなものを括ってみるとよい
    - まずは静的なページを作成してみる
        - 動的にする際は考えることが多様にあるため、まずは静的なページとして作成し、後からコンポーネントやstateを作成して動的にしていく
    - トップダウンとボトムアップ
        - まずどこから作るかが問題になるがレイヤーとして上からやっていくのが簡単なやり方
        - 大きなプロジェクトになるとボトムアップが有効になる
    - stateにするかしないかについて
        - 下記のようなものはstateではない
            - 長らく不変のもの
            - 親コンポーネントから渡されたもの（props）
            - 既存のstateやpropsから導き出せるもの
        - stateをどこで管理するのか
            - stateによってレンダリングされるものを特定
            - レンダリングされる要素の共通の親要素を特定
            - 親要素が見つからなければその要素がstateを管理するのに相応しい
- importとexportについて
    | 構文 | Export 文 | Import 文 |
    | --- | --- | --- |
    | Default | `export default function Button() {}` | `import Button from './Button.js';` |
    | Named | `export function Button() {}` | `import { Button } from './Button.js';` |
- コンポーネントの純粋性
    - 仕事に集中させる（他のオブジェクトや変数を変えない）
    - 同じ入力からは同じ出力
