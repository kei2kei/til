## 勉強になったこと

- React
  - レンダーのタイミング
    - 初回時
        - createRootでrootコンポーネントを作り、それをレンダーする
    - stateの更新時
    - レンダー＝Reactがコンポーネントを呼び出すこと
    - ツリーが上位にあり、下位のコンポーネントを全てレンダーしていくのはパフォーマンスに悪影響を与える可能性がある
  - ReactがDOMへの変更をコミット
      - レンダーが終わるとDOMへのコミットが完了する
      - コミットされたDOMをブラウザが再描画する（ブラウザレンダリング）
  - stateの考え方
      - レンダー＝Reactがコンポーネントをその時々のstateを使ってイベントハンドラなどを作成し、描画するスナップショット
      - その証拠にsetNumber(num+1)を3つ連続させても+1しかしない
          - stateがnum+1になるのは次のレンダリングなので、この瞬間はずっとstateはnumのまま
          - stateが更新されてもコード実行が終わるまではレンダリングには入らない
          - setNumber(num ⇒ num + 1)
              - キュー内の一つ前のstateであるnumに対して+ 1しろという意味
              - 更新用関数
          - stateはミューテート（直接変更）しないでコピーを作成する
          - const a = {…b, a.x = y}
          - const b = {…c, a: a.x = y}
          - ライブラリImmerを使用すると直接変更と同様の変更をしてもコピーが取られる
      - 配列
          - 配列も直接変更はしない
          - 要素
              - 追加(前後)
                  - […array, x]
                  - [x, …array]
              - 削除
                  - filter
              - 変更/置換
                  - map
              - 挿入
                  - […array.slice(0, i), new, …array.slice(i)]
          - コピーしても参照先は同じなので要素の変更はNG
